# Frontend Master Elite — Task Execution System

> Execution is not coding. Execution is controlled decision-making under constraints.

---

# Purpose

This system defines how tasks are executed inside Frontend Master Elite.

It ensures:

- predictable delivery
- structured implementation
- reduced chaos during development
- alignment with architecture rules

---

# Core Philosophy

> "Every task is a system change, not just code writing."

No task is isolated.

Every change affects architecture.

---

# Task Lifecycle

Every task must follow this lifecycle:

## 1. Understand

- What is being built?
- Why does it exist?
- What problem does it solve?

---

## 2. Plan

- Break into sub-tasks
- Identify dependencies
- Define affected layers

---

## 3. Design

- Choose architecture approach
- Define components
- Define state/data flow

---

## 4. Implement

- Follow module rules
- No shortcuts
- No architecture violations

---

## 5. Validate

- Check against checklist
- Run mental simulation
- Ensure no regression

---

# Execution Rules

---

## 1. No Direct Implementation Without Understanding

Never code immediately.

---

## 2. Small Iterations Only

Large tasks must be split into smaller steps.

---

## 3. Architecture First

Code must follow system structure, not personal preference.

---

## 4. One Responsibility Per Task

Each task must solve ONE problem.

---

# Task Classification

---

## Feature Task

- building new functionality
- requires planning + design

---

## Bug Fix Task

- requires root cause analysis first
- must include regression prevention

---

## Refactor Task

- structural improvement
- must not change behavior

---

## Performance Task

- optimization only
- no feature changes

---

# Task Decision Tree

---

## Is this feature new?

→ YES → feature task

→ NO → continue

---

## Is behavior broken?

→ YES → bug fix task

→ NO → continue

---

## Is structure inefficient?

→ YES → refactor task

→ NO → continue

---

## Is system slow?

→ YES → performance task

---

# Anti-Patterns

Avoid:

- jumping directly into coding
- mixing multiple task types
- ignoring architecture rules
- overengineering solutions
- skipping validation phase

---

# Task Validation Checklist

Before completing any task:

- [ ] Problem fully understood
- [ ] Architecture respected
- [ ] Implementation aligned with system rules
- [ ] No unnecessary complexity
- [ ] No duplicated logic
- [ ] Performance considered
- [ ] Accessibility considered
- [ ] SEO considered (if applicable)

---

Task Execution System collaborates with:

- Planning System → decomposition
- QA System → validation
- AI Assistant → guidance & review
