# Frontend Master Elite — Documentation Style Guide

## Purpose

This document defines the writing standards for every Markdown file in the project.

The goal is to ensure:

- Consistency
- Clarity
- Predictability
- Maintainability

Every document must follow this guide.

---

# General Principles

Documentation should be:

- Clear
- Concise
- Technical
- Actionable

Avoid unnecessary explanations.

Avoid marketing language inside technical documents.

---

# Language

All documentation must be written in English.

Use professional technical terminology.

Avoid slang.

Avoid emojis except in README.md.

---

# File Structure

Every steering document must follow this structure.

# Title

Short description.

---

## Purpose

Why this module exists.

---

## When to Use

Explain when Kiro should apply it.

---

## Principles

Core ideas.

---

## Rules

Mandatory rules.

---

## Best Practices

Recommended approaches.

---

## Anti-patterns

Things that must never happen.

---

## Checklist

Validation before finishing.

---

## Examples

Good examples.

Bad examples.

---

## References

Useful links or related modules.

---

# Heading Rules

Use only:

# Level 1

## Level 2

### Level 3

Avoid deeper nesting.

---

# Lists

Prefer bullets.

Example:

- First
- Second
- Third

Avoid long paragraphs.

---

# Code Blocks

Always specify language.

Good:

```ts
const button = true;
```

Bad:

```
const button = true;
```

---

# Tone

Use imperative language.

Good:

"Prefer reusable components."

Bad:

"You should maybe..."

---

# Examples

Every technical rule should include at least one example.

---

# Naming

File names:

lowercase

hyphen-case

Good:

design-system.md

Bad:

DesignSystem.md

---

Every checklist must use checkboxes.

---

# Writing Quality

Every document should answer:

Why?

What?

How?

When?

---

# Final Principle

Every document must teach Kiro how to think, not only what to do.
